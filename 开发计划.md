# xdiag 智能诊断 CLI 工具 - 详细迭代计划

## 迭代策略说明

本迭代计划采用 **增量式开发** 策略，每个迭代都交付可测试、可运行的功能模块。遵循以下原则：

- **MVP 优先**：先实现核心功能，再逐步完善
- **端到端验证**：每个迭代都能完成完整的用户场景
- **自动化测试**：每个功能都有对应的测试方法和验收标准
- **向后兼容**：后续迭代不会破坏已有的功能接口

## 迭代 1：LLM 配置管理 (2-3天)

### 迭代目标
实现完整的 LLM 模型配置管理功能，支持非交互式配置和验证。

### 功能清单
1. **Config Model 命令**
   - 实现 `xdiag config model` 命令
   - 支持必填参数：`--api-key`, `--model-name`
   - 支持可选参数：`--base-url`, `--protocol`
   - 参数验证和错误处理

2. **Config 辅助命令**
   - `xdiag config show`：显示当前配置（隐藏敏感信息）
   - `xdiag config set <key> <value>`：修改单个配置项
   - `xdiag config unset <key>`：删除配置项
   - `xdiag config test`：测试 LLM 连接有效性

3. **环境变量支持**
   - 支持 `XDIAG_API_KEY`, `XDIAG_BASE_URL`, `XDIAG_MODEL_NAME`
   - 配置优先级：环境变量 > 配置文件 > 默认值

4. **Eino 客户端集成**
   - 实现 Eino Client 初始化
   - 封装 LLM 调用接口
   - 错误处理和重试机制

## 迭代 2：目标资产管理 (3-4天)

### 迭代目标
实现完整的诊断目标管理功能，支持多种目标类型和连通性验证。

4. **连通性检测**

## 迭代 3：Playbook 管理系统 (4-5天)

### 迭代目标

实现 Playbook 的加载、管理和执行基础功能，支持分层诊断方案。

### 功能清单

1. **Playbook 目录结构**

   ```
   ./books
   ├── xos-backup
   │   ├── introduction.yaml
   │   └── refs
   │       └── backup-table-lack.yaml
   └── xos-deploy
       ├── introduction.yaml
       └── refs // 以下为具体的book
           ├── deploy-blackbox-failed.yaml
           ├── deploy-k8s-failed.yaml
           └── deploy-xos-agent-failed.yaml
   ```

   - 实现标准 Playbook 目录布局，playbook目录位于~/.xdiag/playbooks
   - 支持 introduction.yaml和 refs/*.yaml
2. **YAML 解析器**

   - 实现 introduction.yaml 结构解析

     - ```
       metadata:
          name: xos安装部署
          desc: 负责解决xos部署时出现问题

       refs:
         - name: deploy_blackbox_failed
           desc: 黑匣子(blackbox)安装失败
           log: 关键日志描述
         - name: deploy_k8s_failed
           desc: k8s安装失败
           log: 关键日志描述
       ```
   - 实现 book 结构解析

     - ```
       name: 黑匣子安装部署诊断
       steps:
         - kind: seq
           desc: 查看节点上/var/xaasos/run.log的相关错误日志
         - kind: branch
           desc: 分析错误日志的内容
           - case: 日志中有黑匣子rpm包已存在并跳过安装的信息
             - kind: seq
               desc: 执行`rpm -e blackbox`卸载黑匣子
             - kind: seq
               desc: 进入/var/xaasos/uninstall目录，执行uninstall_environment.sh
             - kind: seq
               desc: 进入/root/standard-deploy目录，执行./xaasos init --node-ip $node_ip --kubernetes-version 1.25
           - case: 日志中有rsyslog启动失败的信息
             - kind: seq
               desc: 执行systemctl stop rsyslogd 2>dev/null、systemctl stop rsyslog 2>/dev/null
             - kind: seq
               desc: 进入/var/xaasos/uninstall目录，执行uninstall_environment.sh
             - kind: seq
               desc: 进入/root/standard-deploy目录，执行./xaasos init --node-ip $node_ip --kubernetes-version 1.25
       ```
3. **Playbook CLI 命令**

   - `xdiag playbook list `：列出可用 Playbook
   - `xdiag playbook show`：查看 Playbook 详情
   - `xdiag playbook update`：从 GitHub 更新 Playbook
4. **Playbook 注册表**

   - 实现 Playbook 加载
   - 实现标签匹配和过滤

## 风险管理

### 技术风险
1. **Eino 框架集成**：如果 Eino API 有重大变化，需要调整 Agent 实现
2. **LLM 调用成本**：大量诊断可能产生高额 API 费用，需要实现缓存和优化
3. **远程执行安全**：需要严格的安全措施防止恶意命令执行

### 缓解措施
1. **抽象层设计**：为 LLM 和 Agent 框架创建抽象接口，便于替换
2. **成本控制**：实现诊断步骤缓存和智能跳过机制
3. **安全沙箱**：限制可执行命令范围，实施严格的输入验证

### 依赖管理
- **外部依赖**：GitHub API、LLM 服务商 API
- **内部依赖**：各迭代按顺序依赖，但可并行开发非依赖模块
- **回滚策略**：每个迭代都有明确的验收标准，失败时可回滚到上一版本
